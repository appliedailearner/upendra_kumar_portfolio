The Assessment That Wouldn’t Let Go of AVS

A fictional migration story in STAR format about ZionPay Financial Services and the hidden traps in Azure Migrate assessment models.

Upendra Kumar
December 30, 2025
8 min read
Azure Migrate AVS STAR Method

Sometimes the tool is right about the data, but wrong about the destination. This is the story of how a standard "lift and shift" assessment almost derailed a financial services modernization—and how we fixed it.

The Cast

  • Client: ZionPay Financial Services (Fictional)
  • Role: Lead Cloud Architect
  • Challenge: 450 VMs, strict compliance, and an assessment report that insisted on Azure VMware Solution (AVS).

Situation: The "Safe" Bet

ZionPay was running on a legacy vSphere environment that was end-of-support. The directive from the CIO was clear: "Get out of the datacenter by Q3."

The initial Azure Migrate assessment had been run by a partner before I arrived. The report was green across the board. It recommended Azure VMware Solution (AVS) for 95% of the estate. On paper, it made sense. It was the path of least resistance—keep the same hypervisor, keep the same IP addresses, just change the hardware location.

But when I dug into the "Why," I found the trap. The assessment logic had defaulted to AVS simply because the team had selected "Lift and Shift" as the primary goal in the tool settings, and the OS versions were older (Windows 2012 R2). The tool optimized for compatibility, not modernization.

Task: Challenge the Default

My task was to validate if AVS was actually the right financial and technical fit, or if we were about to lock ZionPay into another 3 years of technical debt, just in a different datacenter.

I needed to prove that a mix of Azure Native VMs and Azure SQL Managed Instance would offer better ROI, despite the higher initial migration effort.

Action: The Deep Dive

I initiated a "Level 2" assessment, moving beyond the automated report.

1. The SQL Discovery

The automated tool treated their SQL clusters as just "VMs with high IOPS." I ran a deeper dependency analysis and found that 40% of the SQL workload was actually reporting data that could be easily refactored to Azure SQL Managed Instance.

The Pivot: Moving these to PaaS would eliminate OS patching for nearly half the data estate—a huge win for their small operations team.

2. The Web Front-End Reality

The web servers were stateless. Moving them to AVS (which requires a minimum of 3 nodes) was overkill.

The Pivot: We re-targeted these for Azure Virtual Machine Scale Sets (VMSS). This allowed us to use auto-scaling, paying for only 2 instances at night instead of reserving capacity for peak load 24/7 in AVS.

3. The Cost Comparison

I presented two models to the CIO:

  • Option A (AVS): Fast migration, high fixed monthly cost (3-node minimum), zero modernization.
  • Option B (Native + PaaS): 3-month longer migration timeline, but 40% lower TCO over 3 years and improved security posture.

Result: The Hard Right over the Easy Wrong

The CIO chose Option B.

We didn't just move them; we improved them.

  • Cost Savings: We saved ~$12,000/month by using reserved instances for native VMs and shutting down dev environments at night (which isn't as easily done with the fixed capacity of AVS).
  • Operational Efficiency: The move to Azure SQL MI reduced the patching workload by 15 hours/month.
  • Performance: The application performance improved by 20% because we could size the native VMs to the exact memory requirements, rather than fitting them into the AVS node constraints.

The Lesson

Azure Migrate is a compass, not a GPS. It points North, but it won't tell you about the cliff in the way. Always question the "default" target. The easiest migration path is often the most expensive operational path.
