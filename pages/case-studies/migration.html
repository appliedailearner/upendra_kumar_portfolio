Project Snapshot

This initiative involved the complete retirement of a legacy on-premises data center, migrating 50+ mission-critical VMs to Azure, establishing a hybrid connectivity model via ExpressRoute, and achieving a projected 30% TCO reduction.

Executive Summary

The client, a mid-sized enterprise, was facing a critical "compelling event": their on-premises hardware lease was expiring within 8 months, and the capital expenditure (CapEx) required to refresh standard 3-tier architecture hardware was prohibitive (+40% year-over-year increase).

My role as Lead Architect was to design a lift-and-shift optimization strategy that would not only exit the data center on time but also modernize the operational model from static provisioning to dynamic, cloud-native scalability.

Target Architecture & Connectivity

To ensure a seamless transition with minimal latency, we adopted a classic Hub-and-Spoke network topology.

Hybrid Connectivity (ExpressRoute)

We implemented a 1 Gbps ExpressRoute Circuit with Premium add-on to facilitate global reach if needed later. This dedicated private connection was crucial for:

  • Data Replication: Facilitating the initial bulk data sync of ~20TB of storage without saturating the public internet pipe.
  • Security Compliance: Ensuring database traffic remained on private networks.
  • Latency: Achieving sub-5ms round-trip time (RTT) for hybrid applications during the transitional phase.

The Landing Zone

The target environment was deployed using Terraform (IaC), ensuring that the Virtual Networks (VNETs), Network Security Groups (NSGs), and Azure Firewall policies were immutable and reproducible. We utilized the Cloud Adoption Framework (CAF) Enterprise-Scale architecture patterns.

The Migration Strategy (The 6 R's)

We utilized Azure Migrate as the central orchestration plane. The process was aligned with the industry-standard "6 Rs" framework to ensure every workload found its optimal home.

1. Rehost (Lift & Shift)

For ~60% of the estate (core SQL Server backends and legacy .NET apps), we used ASR (Azure Site Recovery) for block-level replication. This minimized risk and allowed us to seed data over weeks while keeping systems live.

2. Replatform (Tinker & Shift)

Select IIS web front-ends were moved to Azure App Service containers. This provided immediate benefits like auto-patching and slot-swapping without a full code rewrite.

3. Refactor (Re-architect)

We identified a critical inventory processing service that was a bottleneck. We rewrote this component as an Azure Function (Serverless) to handle burst scaling during end-of-month processing.

4. Repurchase (Drop & Shop)

An aging on-premise ticket management system was retired in favor of adopting ServiceNow (SaaS), eliminating maintenance overhead entirely.

5. Retire (Turn Off)

During the assessment phase, we discovered that 15% of the "critical" servers were actually zombies (idle). These were decommissioned immediately, saving estimated $2k/month upfront.

6. Retain (Do Nothing)

A specific mainframe-connected legacy gateway was retained on-premise due to strict latency requirements, connected via the new ExpressRoute circuit.

Technical Challenges Solved

Challenge: Hardcoded IP Addresses

Scenario: Legacy code relied on hardcoded IPs for database connections.
Solution: We implemented an internal DNS Private Zone and used IP forwarding combined with Load Balancer NAT rules to mask the new Azure IPs, allowing applications to function without code rewrites during the migration window.

Final Outcomes

The project was delivered 2 weeks ahead of the hardware lease expiration. Post-migration optimization (sizing down oversised VMs based on Azure Advisor metrics) resulted in an additional 15% monthly saving compared to the initial estimate.

The organization now operates on a fully OpEx model, with improved disaster recovery capabilities (RPO of 15 minutes) and strengthened security posture through Azure Defender for Cloud.
